# Spring Cloud Config Server 도입 요구사항 명세서

- 문서 버전: v0.1
- 작성일: 2026-03-10
- 대상 시스템: `aandiclub.com` 인증 서비스(`auth`) 및 연계 서비스(초기 4~5개, 확장 가능)

## 1. 배경 및 문제 정의
현재 CD 워크플로우(`.github/workflows/cd.yml`)에서 다수의 환경변수를 주입해 Docker Compose로 배포하고 있다.

현 구조의 문제는 다음과 같다.
- 설정값 추가/변경 시 애플리케이션 코드 변경이 없어도 배포 워크플로우를 다시 실행해야 한다.
- 서비스 수(현재 4~5개, 향후 증가)만큼 변수 관리 복잡도와 운영 리스크가 선형 이상으로 증가한다.
- 공통 설정과 서비스별 설정의 책임 경계가 불명확해 변경 영향도 분석이 어렵다.
- 설정 변경의 이력, 승인, 롤백을 표준화하기 어렵다.

## 2. 목표
- 애플리케이션 이미지 재빌드/재배포 없이 설정값을 중앙에서 변경·배포할 수 있어야 한다.
- 여러 서비스의 공통/개별 설정을 표준 구조로 관리할 수 있어야 한다.
- 인증 도메인 보안 기준을 유지하면서 설정 관리 자동화 수준을 높여야 한다.
- 설정 변경 이력 추적, 승인, 롤백을 감사 가능하게 만들어야 한다.

## 3. 범위
### 3.1 포함(In Scope)
- Spring Cloud Config Server 신규 도입
- Config 저장소(별도 Git 저장소) 설계 및 운영 정책 수립
- 클라이언트 서비스(auth 포함)의 Config Client 연동
- 환경(dev/stage/prod) 및 서비스별 설정 계층화
- 설정 변경 반영 프로세스(리프레시/재시작 정책) 수립
- 보안/감사/관측(로그·메트릭·알림) 요구사항 정의

### 3.2 제외(Out of Scope)
- 인증 비즈니스 로직(AuthN/AuthZ) 자체 변경
- 데이터베이스 스키마 마이그레이션 정책 변경
- 서비스 메시/서비스 디스커버리 도입

## 4. 이해관계자
- 서비스 오너(인증)
- 백엔드 개발팀(각 마이크로서비스)
- DevOps/SRE
- 보안 담당자

## 5. 아키텍처 요구사항
- Config Server는 독립 서비스로 운영한다.
- 설정 소스는 `Git(비민감 설정)` + `비밀관리시스템(민감 설정)`의 복합 구조를 기본으로 한다.
- 모든 서비스는 `spring.application.name` + `profile` + `label` 기반으로 설정을 조회해야 한다.
- 공통 설정(`application.*`)과 서비스별 설정(`<service-name>.*`)의 우선순위를 명시적으로 정의해야 한다.
- 운영 환경(`prod`)은 개발/스테이징과 물리 또는 계정 수준으로 분리되어야 한다.

## 6. 기능 요구사항 (FR)
- FR-001: 설정 중앙화
  - 각 서비스는 Config Server에서 런타임 설정을 조회할 수 있어야 한다.
- FR-002: 계층형 설정
  - 공통 설정, 서비스별 설정, 프로파일별 설정, 라벨(브랜치/태그) 기반 설정을 지원해야 한다.
- FR-003: 변경 이력 및 롤백
  - 설정 변경은 Git 이력으로 추적 가능해야 하며, 특정 라벨(태그)로 즉시 롤백할 수 있어야 한다.
- FR-004: 배포 분리
  - 설정 변경 반영을 애플리케이션 이미지 배포 파이프라인과 분리해야 한다.
- FR-005: 변경 반영
  - 설정 변경 후 서비스 반영은 표준화된 방식으로 수행되어야 한다.
  - 초기 단계는 `안전한 리프레시/롤링 재시작` 중 한 방식을 선택해 일관되게 적용해야 한다.
- FR-006: 유효성 검증
  - 필수 설정 누락, 형식 오류, 허용 범위 위반 시 배포 전 검증에서 실패 처리해야 한다.
- FR-007: 서비스 온보딩
  - 신규 서비스 추가 시 표준 템플릿(파일명 규칙, 키 네이밍, 검증 규칙)으로 1회성 설정 등록이 가능해야 한다.
- FR-008: 감사 가능성
  - 누가, 언제, 어떤 키를 변경했는지 추적 가능해야 한다.

## 7. 보안 요구사항 (SR)
- SR-001: 시크릿 분리
  - 비밀번호, 토큰, 키 등 민감값은 Git 평문 저장을 금지한다.
  - 민감값은 AWS Secrets Manager 또는 AWS SSM Parameter Store 등 비밀관리시스템에서 주입해야 한다.
- SR-002: 전송 보안
  - 클라이언트-Config Server 통신은 TLS를 강제해야 한다.
- SR-003: 인증/인가
  - Config Server 접근은 서비스 단위 인증(예: mTLS 또는 서비스 토큰)과 최소권한 정책을 적용해야 한다.
- SR-004: 무결성/신뢰
  - 운영 라벨은 승인된 브랜치/태그만 허용해야 하며, 임의 브랜치 접근을 차단해야 한다.
- SR-005: 로그 보안
  - Config Server 및 클라이언트 로그에 시크릿/PII가 노출되면 안 된다.
- SR-006: Fail-Closed
  - 필수 보안 설정(JWT issuer/audience/secret 등) 조회 실패 시 서비스는 기동 또는 요청 처리를 거부해야 한다.
- SR-007: 키 로테이션
  - JWT/외부 연동 키 로테이션 절차를 문서화하고 무중단 또는 최소중단으로 수행 가능해야 한다.

## 8. 비기능 요구사항 (NFR)
- NFR-001: 가용성
  - Config Server는 단일 장애점(SPOF)이 되면 안 되며, 운영 환경에서 다중 인스턴스 구성을 지원해야 한다.
- NFR-002: 성능
  - 설정 조회 API의 p95 응답시간은 500ms 이하여야 한다(정상 부하 기준).
- NFR-003: 확장성
  - 초기 5개 서비스, 향후 20개 서비스까지 설정 조회/반영 운영이 가능해야 한다.
- NFR-004: 복구
  - 설정 저장소 장애/오염 시 이전 정상 라벨로 신속 롤백할 수 있어야 한다.
- NFR-005: 관측성
  - 설정 조회 실패율, 리프레시 성공/실패, 라벨 변경 이벤트를 메트릭/로그로 수집해야 한다.

## 9. 설정 모델 표준
- 파일 네이밍
  - `application.yml` (공통)
  - `application-<profile>.yml` (환경 공통)
  - `<service-name>.yml` (서비스 공통)
  - `<service-name>-<profile>.yml` (서비스+환경)
- 키 네이밍
  - 도메인 prefix를 명시한다. 예: `security.jwt.*`, `app.invite.*`
- 기본값 정책
  - 보안 민감 키에는 코드 기본값을 두지 않는다.
  - 운영 필수 키 누락 시 기동 실패를 기본 정책으로 한다.

## 10. 운영 프로세스 요구사항
- 설정 변경은 PR 기반 승인 절차를 거쳐야 한다.
- 운영 반영은 `config-repo` 머지 이벤트로 트리거되어야 한다.
- 변경 후 자동 검증(스키마/필수 키/형식 검사)을 통과해야 반영된다.
- 반영 실패 시 자동 롤백 또는 즉시 롤백 가이드를 제공해야 한다.

## 11. 수용 기준 (Acceptance Criteria)
- AC-001: 설정값 1개 변경 시 애플리케이션 이미지 재빌드 없이 운영 반영이 가능해야 한다.
- AC-002: 신규 설정 키 1개 추가 시 기존 CD 전체 재실행 없이 반영 경로가 제공되어야 한다.
- AC-003: 잘못된 설정(필수 키 누락/형식 오류)은 사전 검증 단계에서 차단되어야 한다.
- AC-004: `auth` 포함 5개 서비스에서 공통/개별 설정 조회가 정상 동작해야 한다.
- AC-005: 운영 시크릿은 Git에 저장되지 않으며, 로그에도 노출되지 않아야 한다.
- AC-006: 운영 환경에서 라벨 기반 롤백을 10분 이내 수행할 수 있어야 한다.

## 12. 마이그레이션 요구사항
- 1단계: 현재 `cd.yml` 환경변수 목록을 분류한다.
  - 비민감 설정: Config Repo로 이동
  - 민감 설정: 비밀관리시스템으로 이동
- 2단계: `auth` 서비스에 Config Client를 우선 적용한다.
- 3단계: `auth` 운영 안정화 후 타 서비스로 점진 확산한다.
- 4단계: 기존 CD의 대량 env 주입 로직을 축소/정리한다.

## 13. 리스크 및 대응
- 리스크: 잘못된 공통 설정이 다수 서비스 장애를 유발할 수 있음
  - 대응: 환경별 승인 게이트, 카나리 반영, 즉시 롤백 라벨 운영
- 리스크: 시크릿 분리 미흡 시 보안사고 발생 가능
  - 대응: Git 평문 시크릿 금지 정책 + 스캔 도구 + 리뷰 체크리스트 강제
- 리스크: 런타임 리프레시로 인한 예측 불가능성
  - 대응: 리프레시 허용 키 allowlist 운영, 고위험 키는 롤링 재시작만 허용

## 14. 의사결정 필요 항목
- Config Server 인증 방식: `mTLS` vs `서비스 토큰`
- 변경 반영 방식: `Spring Cloud Bus` vs `오케스트레이션 기반 /actuator/refresh`
- 비밀관리 백엔드: `AWS Secrets Manager` vs `AWS SSM Parameter Store`
- Config Repo 운영 전략: `환경별 브랜치` vs `단일 브랜치 + 라벨/경로 분리`

## 15. 완료 정의 (Definition of Done)
- 요구사항 문서 승인
- Config Server 운영 배포 완료(dev/stage/prod)
- `auth` 서비스 연동 및 회귀 테스트 통과
- 보안 점검(시크릿 노출/접근통제/로그 마스킹) 완료
- 운영 런북(장애 대응, 롤백, 키 로테이션) 문서화 완료
